## 一、 Python底层与并发

### O1. 介绍一下 Python 中的 GIL？它对多线程的影响以及如何避开？
**👎 普通回答：**
> GIL 是全局解释器锁。因为有它，Python 的多线程是假的多线程，一次只能跑一个个任务。所以想要真正的并发，得改用多进程，或者用其他语言写。

**👍 满分范例：**
> “GIL 是 CPython 解释器引擎历史遗留的设计。为了防止多线程并发时修改对象的**引用计数（Reference Count，Python GC机制的核心）**发生抢占而导致内存泄漏崩溃，强制要求执行任何 Python 字节码的线程必须先获取这把互斥锁。
> **影响分场景**：
> 1. 如果是 **CPU密集型任务**（如大规模矩阵计算、图像处理），Python 的多线程不仅无法并发，甚至因为不断地发生 GIL 锁释放抢占的上下文切换开销，比单线程跑得还要慢。
> 2. 如果是 **I/O密集型任务**（如网络爬虫、文件读取），在遇到 `time.sleep` 或 Socket 阻塞时，线程会自动释放 GIL，此时多线程依然能发挥非常卓越的并发优势。
> **如何规避**：
> 在AI处理中，要真正拉起多核算力应对CPU密集计算，我们会：1) 使用 `multiprocessing` 开启纯多进程，每个进程独立占用一个Python解释器与内存空间避免由于GIL互相卡脖子；2) 将核心计算转交给 C/C++ 扩展（如 Numpy、Cython 底层），在进入 C 层时释放 GIL 锁； 3) 引入 `asyncio` 协程进行极限 IO 压榨。”

**💡 得分点剖析：**
- 直击痛点：交代 GIL 的存在是为了保护 CPython 下的 **引用计数(GC)**。
- 立刻划分场景，澄清并非“所有多线程都废”，说明 I/O 阻塞会自释放 GIL。
- 给出多进程与 C底层计算拓展转移的正确解法。

## 二、 深度学习与大模型底层架构

### O2. 详细讲解 Transformer 中的 Self-Attention 为什么还要除以一个缩放因子 $\sqrt{d_{k}}$（根号d）？如果不除表现会怎样？
**👎 普通回答：**
> 为了缩放数值，如果不除这一个数的话，Attention 得分就会变得非常大，导致模型收敛不好或者梯度消失，训练崩掉。

**👍 满分范例：**
> “这是为了把数据方差拉平，防止激活函数饱和陷入停滞死区。
> 我们知道，Query(Q) 和 Key(K) 分别是经过参数矩阵投影的特征向量序列。假设 Q 和 K 内部元素的均值为 0，方差为 1 且互相独立。那么当他俩进行内积（点乘）计算产生一个数值时，由于内积是对应元素相乘再累加，结果这一个点积结果的方差就从 1 被狂飙放大到了空间的维度 $d_k$（例如 64 或者 128）。
> 这样巨大的方差会导致 Q和K点积后的数值两极分化极为严重，有的极大有的极小。
> 接下来内积结果要进入 Softmax() 函数。而 Softmax 在输入值极端巨大或微小时，曲线会立刻变得极度平坦——也就是落在饱和区。在反向传播时，平直区域的导数近乎为 0。这就直接引发了非常恐怖的梯度消失现象，参数彻底不更新了。
> 为了强行制裁方差突变，论文用除以标准差（也就是方差的开根号 $\sqrt{d_k}$）的方式强行将内积结果方差拉回 1，让送入 Softmax 的数字全部聚焦在导数最饱满敏感的陡峭梯队上，从而保护了模型正常收敛。”

**💡 得分点剖析：**
- 从统计学角度点破：内积导致方差直接爆棚扩张到 $d_k$。
- 指出高方差会把数值推入 Softmax 死区（Saturation 区），导数为0。
- 解释根号相当于做除了**标准差的归一化平抑动作**，保证反向传播链存活。

## 三、 大模型微调与工程拉平

### O3. LoRA 微调的核心思想是什么？相较于全量参数微调优势在哪？什么是 QLoRA？
**👎 普通回答：**
> 全量微调参数太多显卡装不下。LoRA 就是把原模型冻结不开，在旁边挂两个小一点的网络来训练参数，这样参数量就变小了。QLoRA 则是更小，用了量化手段。

**👍 满分范例：**
> “LoRA 建立在一个学术基本假设上：即大语言模型虽然庞大，但其在特定子任务做领域适配适应时，**权重更新的变化量存在着极强的低秩（Low-Rank）特性**。
> **核心机制**：在微调时，LoRA 完全锁死预训练好的主基座权重矩阵 $W$，然后在旁边并行挂载一个支路。这个支路是由两个降维投影小矩阵 $A$（列数小，负责降维）和 $B$（行数小，负责恢复升维）构成。前向推导时，最终结果是 $h = Wx + BAx$。
> **优势与合并**：这使得仅有一万分之一到百分之一的参数需要记录梯度进行更新计算，完全抹平了优化器状态与巨量梯度占用掉的多卡显存。而且在推理部署时，我们可以直接把矩阵相乘的常数结果加进 $W$ 里面去（$W' = W+BA$），实现推理延迟 0 增加。
> **QLoRA**：在 LoRA 基础上直接向加载原模型开刀。QLoRA 会把极其庞大的底座模型直接经过 NF4（4-bit正态浮点数）进行压榨级别的量化并使用 Paged Optimizer 解决尖峰显存，使得原本需要四张A100的 65B 模型变成能够塞进单张普通显卡训练，而它的训练层依然保持原精度进行保障模型智商损失极微。”

**💡 得分点剖析：**
- 精确提出 **低秩(Low-Rank)** 假说与 **$A$降秩/$B$升秩** 旁接矩阵算子。
- 提到训练完成后可融回 $W$ 原矩阵，**不影响推理耗时**的高级工程视野。
- 说明 QLoRA 是针对底层模型做了极细化的 **4-bit NF4 重度量化加载**。

## 四、 风控/结构化场景 (场景防坑)

### O4. 在进行业务训练（如极罕有欺诈监测）时，碰到了模型预测“正例”（也就是黑产）的准确率很高其实是由于极端不平衡样本导致的（正例1：负例99）。你如何解决样本不平衡引发的模型失效？
**👎 普通回答：**
> 样本不平衡会导致模型都偏向于猜正常用户。我可以通过上采样把黑产数据拷贝几十遍匹配上正常数据的数额，然后训练模型。最后换一种评价指标看。

**👍 满分范例：**
> “简单的复制上采样(Over-Sampling)极度容易导致模型对少数样本陷入严重过拟合。面对诸如1:99的长尾极端分布，我采取体系化的三套拳。
> 1. **数据与采样面**：如果是传统树模型，抛弃机械复制改用 **SMOTE / ADASYN 等启发式插值算法**，通过观察少数派近邻规律人工生成有边界偏转的伪数据强行补足数据面。如果是下采样(Under-sampling)健康样本，需要组装多个下采样的子集用集成模型去做综合（如 EasyEnsemble）。
> 2. **损失函数重写（核心面）**：普通交叉熵在面对极端分布时，因为健康的大量负例提供的损失值能全盘淹没掉少数的正例，模型会直接摆烂躺平。需要**引入 Focal Loss 焦点损失或 Cost-Sensitive 的带权损失**。给判错极度稀缺的正样本非常恐怖的高额罚款权重$(w)$，同时引入衰减项 $(1-P)^\gamma$ 让模型别管那些轻易就能分清的大量多数类，全力盯紧模糊的临界少数类。
> 3. **验收评估指标重抛**：全面无视毫无意义的 Accuracy，而是引入 **PR曲线 (查准率/查全率)** 及 **ROC-AUC (受试者工作特征曲线面积)** 来准确评估模型在极度不平衡数据下的真实分类能力。”

**💡 得分点剖析：**
- 不仅仅答出上/下采样，而是带入了更高级的插值生成器 **SMOTE 算法**。
- 深刻认知到由于样本失衡所以传统损失函数会摆烂的原因，提议通过 **Focal Loss** 对难分长尾样本进行重度惩罚兜底。
- 定局提出舍弃高迷惑性 Accuracy 改用 **Recall回调 / F1 / AUC** 面积验收。
